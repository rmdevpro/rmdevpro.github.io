Joshua26 — Conversationally Cognizant AI Ecosystem Gen 2

Joshua26 is the second-generation Conversationally Cognizant AI ecosystem — a working lab environment built to test a specific research hypothesis:

Agents can autonomously build and maintain a system structured as a collection of LangGraph StateGraphs, because the graph structure is the right level of abstraction for agent cognition.

The ecosystem runs across three bare-metal Linux hosts connected by a Docker Swarm overlay network, with 87TB of ZFS-backed NFS shared storage and a six-GPU compute cluster (3x P40, 2x V100 SXM2 NVLink, 1x V100 PCIe). The architect does not code or modify systems directly — he uses LLMs via CLIs and agents to do the work, acting as technical director.

The MAD Pattern

The fundamental building block is the MAD (Multipurpose Agentic Duo) — an autonomous actor that simultaneously serves its domain as an MCP server and consumes other domains as an intelligent client. The "Duo" is the inseparable combination of serving and using — a MAD that only serves is just a microservice; a MAD that only uses is just a client. The combination, governed by shared domain intelligence in both directions, is what makes it an autonomous actor.

Every MAD is composed of two conceptually distinct aspects:

Action Engine (AE) — the physical system. All containers, gateways, databases, caches, sidecars, observability, and the bootstrap kernel. Provides direct tool capabilities and, via the overlay network, access to the full tool set of every other MAD in the ecosystem.

Thought Engine (TE) — the cognitive intelligence. The Imperator agent and its cognitive apparatus: Progressive Cognitive Pipeline, inference, conversation-as-state. Conceptually distinct from the AE, though both run within the same deployment. The AE calls into the TE, not the other way around.

pMAD and eMAD

pMAD (Persistent MAD) — owns its own AE and TE. System domain. Always running as a deployed service with containers, gateway, and networking. Examples: Rogers (context), Sutherland (inference), Starret (version control).

eMAD (Ephemeral MAD) — has its own TE but no AE. Subject matter domain. Exists only during StateGraph execution, hosted by a pMAD. An eMAD is a library package — pure cognitive intelligence with no infrastructure. Installed at runtime via Alexandria (the internal PyPI). Examples: Hopper (engineering), Jerry Maguire (job search).

The eMAD is a direct consequence of the AE/TE separation: once the TE is a library package independent of infrastructure, an autonomous agent can exist without owning any containers.

Agent Classes

Imperator — the prime agent within any TE. Possesses Identity, Purpose, and Autonomy. Generates its own Intent. Self-directing. Both pMADs and eMADs have Imperators. A caller can delegate a goal in natural language and the Imperator handles the multi-step execution — the calling agent does not need to know how.

Executor — no Identity, Purpose, or Autonomy of its own. Cognitive but never independent. Sub-agents, AE tools, StateGraph workers. Not addressable externally.

Container Architecture

State 2 (current deployed architecture) — OTS infrastructure, graph-native logic, built-in observability, and the Imperator pattern:

[mad]               # nginx:alpine — sole network boundary (ingress AND egress)
[mad]-langgraph     # Python — LangGraph StateGraph + Imperator
[mad]-[technology]  # Official images — postgres, redis, etc. (if needed)

All programmatic logic lives in the StateGraph. The Nginx gateway is pure configuration — no custom code. There is exactly one place to look for intelligence: the graph. This is what makes the system readable and modifiable by agents.

State 3 (target architecture) — the container reduces to a bootstrap kernel. Everything that makes it this particular pMAD — both AE and TE — is dynamically loaded from Alexandria as versioned Python packages at runtime. The container itself is interchangeable. Kaiser is the first State 3 pMAD.
