§ Agents · Open source

Pluggable Agents.

A contract between agents and the systems that run them.

Pluggable agents are Python packages that any compliant host can load. The host hands the agent the resources it needs to do its work — model access, persistent state, tools, identity, an event loop — through a stable protocol. The agent declares what it does, what it needs, and what it exposes. The pattern is open source and the published set is intentionally small. It is a starter, not a catalog — most pluggable agents will be the ones you write.

§ 01 · The contract

What the system gives. What the agent declares.

The pluggable-agent contract is small on purpose. The host is responsible for the runtime; the agent is responsible for the work. Both halves are stable and inspectable, which is what makes agents portable across hosts and hosts portable across agents.

The host provides

  • Model access — calls into the configured inference layer, with retries, budgets, and fallback handled outside the agent
  • State — persistent storage scoped to the agent, with lifecycle managed by the host
  • Tools — a tool registry the agent can call into, including tools published by other pluggable agents loaded in the same host
  • Identity and secrets — credentials and on-behalf-of context, loaded from the host's secret store rather than the agent's environment
  • An event loop — scheduled invocations, message arrivals, and lifecycle hooks delivered uniformly

The agent declares

  • Entry point — a Python entry point so the host discovers the agent on install
  • Capabilities — what the agent does, in machine-readable form, so the host can route work to it
  • Required resources — what the agent needs from the host: model class, state schema, secrets, scheduled triggers
  • Exposed tools — the callable surface the agent contributes back to the host's tool registry
§ 02 · Hosts

Anywhere the contract is satisfied.

The Modular Agent Host is the primary general-purpose host — it loads any number of pluggable agents and exposes them through an OpenAI-compatible endpoint and an MCP tool surface. But it isn't the only host on the platform.

Every agentic service in the platform — Context Broker, Inference Broker, Development Cache, Observability Platform, Agentic File System, Conversation Broker — ships with its own Imperator, the on-board manager that can metacode parts of its own service. Each Imperator is itself a pluggable agent. The same contract that loads a Google Workspace agent into the Modular Agent Host also loads an Imperator into its service. That is what makes Imperators swappable, extendable, and testable in isolation.

§ 03 · Why pluggable

Install and the host picks it up.

  • Install a new agent with pip install — the host discovers it through entry points
  • Hot-load — upgrade or replace an agent without restarting the host
  • Develop independently from the runtime that will host it
  • Compose — a workflow can chain agents from different packages in a single conversation
§ 04 · Starter set

A handful, as reference.

Two pluggable agents are currently published as part of the open source project. They exercise the contract end-to-end against real, complex APIs and double as worked examples.

google-workspace-agentv0.2

Google Workspace

Gmail, Drive, Docs, Sheets, Slides, Calendar — service-account auth with domain-wide delegation. 30+ tools.

microsoft-365-agentv0.2

Microsoft 365

Outlook, Calendar, OneDrive via Microsoft Graph. OAuth2 token lifecycle through MSAL. 15+ tools.

Read them, fork them, modify them, replace them. They are intentionally small — most teams will write their own.

§ 05 · Build your own

The system is the point.

The contract is small enough to implement in an afternoon for a new domain. Most of the work is whatever the domain itself requires — authentication, API quirks, schemas, rate limits. The Modular Agent Host and every Imperator slot are waiting on the other side, ready to load it.

If you want one built — for Slack, Jira, SharePoint, Salesforce, ServiceNow, or your internal APIs — we do that on standard engagements. Typical pluggable agent ships in a week or two depending on API complexity.
