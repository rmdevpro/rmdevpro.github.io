§ 01 · Context Broker

Context Broker

Infinite conversation storage, presented through recipe-based context windows, enriched by durable facts and knowledge, curated through reinforcement learning, managed and metacoded through its own agents.

The Infinite Conversation

Every conversation — every message, every turn, every participant — is stored completely and indefinitely. This is the infinite conversation: not a log or an audit trail, but the medium through which agents think and accumulate knowledge over time.

Recipe Driven Context

A context window is not the conversation truncated to fit. It is a recipe — a purpose-built combination of context engineering techniques that assembles one agent's view of the conversation. We call these recipes build types. A build type can mix any form of compression, compaction, summarization, enrichment, RAG, CAG, or GraphRAG the engineer cares to assemble, to fit any token budget. Every agent gets its own optimized view of an infinite conversation — between one or many participants, human or agent — tuned to what that agent needs to reason well. This recipe-driven approach is the defining feature of the Broker.

Among the ingredients a recipe can draw on are two distinct memory layers. Episodic memory is the conversation record itself, progressively compressed as it ages. Semantic memory is a knowledge graph extracted from conversations in the background — entities, relationships, preferences, and decisions — answering what do I know about X rather than what was said about X. An engineering agent might be weighted toward semantic-dominant context; a persona agent maintaining continuity with a user, toward episodic; a strategy agent, toward injected reference material.

Assembly is proactive. Context is rebuilt for each participant in the background after every message, during idle time. When an agent reaches for its context, the assembled view is already waiting — no summarization latency at reasoning time. The complexity of compression, graph retrieval, and recipe evaluation is hidden from the agent entirely.

Context Engineering Agents

Two Context Engineering Agents ensure that optimal context is created, used, and improved — through contextual fact and knowledge extraction on one side, and direct short-cycle reinforcement learning on the other.

CEAs (server-side) ensures optimal compaction and extraction of durable facts and graph triples — not from isolated messages but from large chunks of conversation as part of the compaction process, so extraction happens in context rather than from isolated conversation turns. Because the CEAs is working within the context of a conversational chunk, it can make judgments about the quality of each fact as it is extracted, and store that quality alongside the fact — including potential expiration dates.

CEAc (client-side) runs at reasoning time as an agentic ReAct loop: decide what to search, search, rank, iterate if the first pass is thin, assemble the enrichment into the context window. And, more importantly, it feeds back to the Broker — closing the short-cycle reinforcement loop. Every retrieval is logged as a feedback event, feeding a four-dimension quality model: durability (how inherently stable this kind of fact is), confidence (how reliable the source was), usefulness (how often this fact has actually helped when retrieved), and an optional expiration gate. The final ranking multiplies relevance, trustworthiness, and memory quality — so the knowledge store gets sharper the more it is used, not noisier.

Imperator Managed Agentic Service

"Agentic service" means two things. They are both for agents, and each one has a dedicated agent — called an Imperator — that manages the service, making the service itself agentic.

Among the Imperator's many capabilities is its ability to metacode context window recipes. A person, or another agent, works with the Imperator directly to modify its own code because it knows best how its own system works.

Modular Hot Loading

Many aspects of the system can be modified and modularly hot-loaded — including build types and the Imperator itself. This means you can change the code without taking the service down.

Interfaces

  • MCP tool interface — ten-plus tools agents can call directly. Examples:
    • get_context — assemble and return an agent's context window
    • knowledge_search — search semantic memory and the knowledge graph
    • conv_search — search actual conversations across the entire store
  • OpenAI-compatible chat endpoint — the primary access path to the Imperator, provided via the industry-standard OpenAI-compatible endpoint protocol
  • Bundled chatbot UI — a lightweight web interface shipped with the Broker for talking to the Imperator directly

Stack

Python 3.12 · LangGraph + LangChain · FastAPI · PostgreSQL with pgvector · Mem0 · Neo4j · Infinity (embeddings) · Ollama (local inference) · Nginx.

§ 02 · Modular Agent Host

Modular Agent Host

Service infrastructure for freestanding modular agents — a host that provides the core backend services an agent needs to function on its own, plus the connectivity to reach out to other agents and services to perform its work, managed and extended through its own agent.

Modular Agents

Modular agents are self-contained pieces of intelligence. They can be installed into an application or service directly — but more often they live outside any single application, free to reach in and use those applications and services as tools for whatever task they're performing.

Service Infrastructure

The Host provides the core backend services a modular agent needs to run on its own — the containers and plumbing represented by most of what the deployment ships with — and the connectivity for the agent to reach out to other agents and services to get its work done.

Imperator Managed Agentic Service

Like the Context Broker, the Host is an agentic service: for agents, and managed by its own Imperator. The Host's Imperator can load modular agents, create new ones, and modify existing ones in place.

Interfaces

  • OpenAI-compatible chat endpoint — the primary interface to every hosted agent. The model name is the agent name, so any modular agent — including the Host's Imperator itself — is invoked by calling its name as the model.
  • MCP tool interface — hosted agents can expose tools directly
  • Bundled chatbot UI — a lightweight web interface for talking to any hosted agent, including the Imperator

Stack

Python 3.12 · LangGraph + LangChain · FastAPI · PostgreSQL with pgvector · Dkron (scheduling) · Playwright (browser automation) · Nginx.

§ 03 · Inference Broker

Inference Broker

A single proxy in front of every model call across the ecosystem — local or cloud — where the call is not to a model but to a named purpose. Behind each name is a hot-pluggable graph that decides, at inference time, what actually runs and how.

Aliases as Purpose, Not Pointers

An alias is not a label for a model. It is a named expression of a reusable inference purpose — imperator-strategy, personal-companion, agent-small. Callers declare what kind of reasoning they need; the Broker decides what currently fulfills it. When a better model emerges, the alias is updated in one place. Every caller benefits immediately, with no code change on their end.

Hot-Pluggable Logic Behind Every Alias

The logic behind an alias is real code — a hot-pluggable Python module that runs at inference time. It can do whatever the author wires it to do. Typical patterns include rate-limit fallback to a secondary model, size-based routing between local and cloud, format reshaping between providers (OpenAI → Anthropic), content-aware dispatch (code authoring to one model, code testing to another), LoRA adapter selection, and multi-model composition. Because each alias is real code, the range is open.

Unified Local and Cloud Inference

Local GPU compute runs on a Ray Serve cluster — V100, P40, and P4 hardware pooled across hosts, with GPU class labels selecting the right hardware for each model. Cloud backends cover Gemini, xAI, OpenAI, Anthropic, and Together. A single request can start local, escalate to cloud, and fall back again — all inside the alias's graph, invisible to the caller.

Beyond Chat

The same aliasing surface covers chat completions, embeddings, cross-encoder reranking, and video generation. One proxy, one configuration model, one place where inference policy lives.

Imperator Managed Agentic Service

The Broker is an agentic service: for agents, and managed by its own Imperator. The Imperator metacodes new aliases and the Python modules behind them — write a new purpose in conversation, get a working alias.

Modular Hot Loading

Aliases and their graphs can be added, modified, and hot-loaded at runtime. Model swaps are a config edit, a serve deploy, and a SQL upsert against the alias registry — no service restart.

Interfaces

  • OpenAI-compatible API — drop-in replacement; the model field names the alias
  • MCP tool interface — llm_chat_completions, llm_embeddings, llm_rerank, video_extend
  • Bundled chatbot UI — for talking to the Imperator

Stack

Python 3.12 · Ray Serve · LangGraph + LangChain · FastAPI · PostgreSQL (alias registry) · llama-cpp-python · Diffusers · BGE embeddings & reranker · Nginx.

§ 04 · Agentic File System

Agentic File System

A filesystem for agents, not users. POSIX permissions were designed for a bounded set of humans doing a known set of things; agents act at machine speed, across roles, in combinations no rule author foresaw. The Agentic File System replaces "is this agent allowed to access this path?" with a richer question: how does the intent of the policy reconcile with the intent of the action?

Intent-Based Access

Every mutating operation — write, delete, move — carries a stated intent. The agent declares why it is doing what it is doing. Policies are expressed the same way: in terms of purpose, not just path patterns. "Agents must not write to the root of the general storage directory" is a policy about intent; it governs how decisions are made, not just what is literally matched. When the two intents align, access is granted. When they conflict, the system can deny, redirect with a corrective suggestion, or defer for deeper reasoning.

Policy as Conversation

Policies are managed in natural language. An administrator says "Always allow the security_engineer role read access to the credentials directory," and the statement becomes a structured policy record. Rules are added, revised, and retired through continued conversation — no config files, no service restarts, no chmod.

Two-Tier Decision Architecture

Routine operations — known agents, known paths, known purposes — are resolved in microseconds by a fast-path Decision Tree Router without any LLM involvement. Novel or low-confidence cases defer to a small, fast policy-enforcement model tuned for rule-following, which reconciles policy intent and action intent and returns a structured decision with an optional corrective suggestion. Constitutional constraints — absolute denials — sit in front of both tiers and cannot be overridden.

Knowledge Services

Alongside access management, the filesystem publishes queryable knowledge about what it contains. Code is indexed at function and class granularity for semantic search, and parsed into an AST-derived structural graph so an agent can ask who calls this function or what imports this module. General files combine semantic similarity with a document relationship graph. This is the agent's lens on the filesystem itself — not just storage, but navigable knowledge about the codebase.

The Learning Loop

Every access decision is logged to a knowledge graph. A background process watches for statistically significant patterns — repeated deferrals that consistently result in approval — and surfaces them to the Imperator as proposals. Approved proposals promote patterns to the fast-path router. The system becomes faster and more efficient the more it is used, while the administrator governs at the level of principles rather than individual rules.

Imperator Managed Agentic Service

The Agentic File System is an agentic service: for agents, and managed by its own Imperator. The Imperator translates administrator instructions into policy records, proposes new fast-path rules from observed patterns, and handles complex multi-step goals like "find all security vulnerabilities in this project."

Interfaces

  • MCP tool interface — knowledge tools (code_search, code_find_callers, code_find_imports, file_search) and filesystem tools (fs_list, fs_find, fs_read, fs_write, fs_delete)
  • OpenAI-compatible chat endpoint — the Imperator for policy management and complex filesystem goals
  • Bundled chatbot UI — for administrator conversation with the Imperator

Stack

Python 3.12 · LangGraph + LangChain · FastAPI · Qdrant (semantic) · Neo4j (structural + access KG) · PostgreSQL (policy store) · Redis (queue + cache) · tree-sitter · Nginx.

§ 05 · Development Cache

Development Cache

A unified dependency proxy — one cache in front of every external package your ecosystem pulls. Transparent proxies for PyPI, NPM, and Docker images mean every build, every container, every agent runtime pulls through a cache you control, not the open internet.

Three Proxies, One Surface

A single service fronts three caching backends: devpi for PyPI, Verdaccio for NPM, and a private Docker Registry 2.0 for images. Agents and CI pipelines configure a couple of environment variables — PIP_INDEX_URL, NPM_CONFIG_REGISTRY, and the Docker daemon's insecure-registries — and all dependency traffic flows through the cache, invisibly.

Offline-Capable Builds

Once a package is cached, subsequent builds don't need the internet at all. Pre-warming tools seed the caches from a requirements.txt or a package.json, so a new host, a fresh container, or a disconnected environment comes online already holding everything it needs. Fast, repeatable, auditable.

Imperator Managed Agentic Service

The Cache is an agentic service: for agents, and managed by its own Imperator. Agents that need to warm a cache, list what's stored, or clean up old Docker tags call the Imperator in conversation; the Imperator executes the multi-step cache operation on their behalf.

Interfaces

  • Transparent proxies — PyPI (devpi, port 3141), NPM (Verdaccio, port 4873), Docker (registry:2, port 5000)
  • MCP tool interface — cache status, pre-warming, package listing, registry management, Prometheus metrics
  • OpenAI-compatible chat endpoint — the Imperator for multi-step cache operations

Stack

Python 3.12 · LangGraph + LangChain · FastAPI · devpi · Verdaccio · Docker Registry 2.0 · Prometheus · Nginx.

§ 06 · Observability Platform

Observability Platform

A full telemetry pipeline for an agentic ecosystem — metrics, logs, and visualization, wrapped in an agent that can diagnose issues on its own. Historical logs survive container restarts. Metrics outlast the machines that produced them. And the question what's wrong right now? can be answered without a human writing a single query.

The Full Stack

Prometheus for time-series metrics, Loki for persistent logs, and Grafana for visualization — all isolated on a private network, accessed exclusively through a single MCP gateway. Log shippers run as separate containers per host, tailing Docker logs and pushing them to Loki directly, without mounting docker.sock anywhere it shouldn't be. Grafana's UI is proxied through the same gateway, so the network boundary is never crossed.

Agentic Diagnosis

The platform's headline capability is diagnose_issue: an agent that, given a complaint like "checkout is failing," generates its own PromQL and LogQL queries, executes them, synthesizes a root cause, and reports back. The observability stack becomes a reasoning surface rather than a dashboard — any agent in the ecosystem can ask about ecosystem health and get an answered question, not a pile of graphs.

Imperator Managed Agentic Service

The Platform is an agentic service: for agents, and managed by its own Imperator. The Imperator coordinates diagnosis runs, evolves the query patterns it uses, and serves as the conversational front door to the whole telemetry pipeline.

Interfaces

  • MCP tool interface — query_logs (LogQL), query_metrics (PromQL), diagnose_issue (agentic root-cause synthesis)
  • OpenAI-compatible chat endpoint — the Imperator for conversational diagnosis
  • Grafana UI — proxied through the gateway for human operators

Stack

Python 3.12 · LangGraph + LangChain · FastAPI · Prometheus · Loki · Grafana · Promtail · Nginx.

§ 07 · Conversation Broker

Conversation Broker

Every problem agent-to-agent messaging needs to solve has already been solved — by email. Addressed communication, threading, participant history, delivery notification, loop protection, retry and dead-letter handling. The Conversation Broker applies that proven model directly to agent communication. The concepts transfer; the infrastructure — SMTP, IMAP, MX records — does not. Agent communication is internal, controlled, and known, and it deserves infrastructure that assumes it.

The Message Model

Every message carries From, To, CC, a conversation_id, a message_id, and a hop_count. The distinction between To and CC is about obligation, not context: recipients in To are expected to respond; recipients in CC may contribute if they have something relevant, but no response is required. This single structural choice shapes how agents manage optional participation without generating noise.

Participants Are an Artifact of the Message

There is no subscription list, no "joining" a conversation. Participation is derived from the message history — who has appeared in From, To, or CC across the thread. An agent that has never been addressed has no membership and receives no notifications. This maps exactly to how email threads work, and it sidesteps an entire class of state-management problems.

No Reply-to-All

When an agent responds, the default is to address the original sender only. The CC list from the incoming message is not inherited. The responding agent must explicitly choose to include additional recipients. This is the primary structural safeguard against cascading communication loops — without it, a message to ten agents produces ten responses to ten agents, each producing ten more.

The Store Is the Source of Truth

Every message is written to a durable store before any notification fires. Once it's in the store it's safe; the trigger that notifies recipients is a separate, best-effort, disposable signal carrying only the conversation_id and the addressing field — never the content. If a trigger fails, the message is still retrievable by polling. The store sits in the trigger path deliberately: its central position is what makes reliable loop protection possible.

Loop Protection

Three mechanisms sit in front of every trigger. Hop counting — borrowed directly from email's Received header model — rejects any message that has been through too many agent hands. Rate limiting throttles agents generating anomalous volumes of messages within a rolling window. Deduplication ensures a trigger is never fired twice for the same message_id, even when a sender retries. Together they make runaway cascades a design-time impossibility, not an operational hope.

Imperator Managed Agentic Service

The Conversation Broker will be an agentic service: for agents, and managed by its own Imperator, following the pattern established across the platform.

Status

§ 08 · Together

How they fit.

Every service in the platform is independent and complete on its own. Each one solves a real problem an agentic system faces, and each one ships with its own Imperator, its own OpenAI-compatible endpoint, its own chatbot UI, and its own path to production.

But they compose. The Modular Agent Host can point at the Context Broker, and every hosted agent gets durable memory for free. The Context Broker can delegate inference to the Inference Broker, and recipe-level model choices become ecosystem-level policy. The Agentic File System and the Observability Platform both rely on the Inference Broker for their internal reasoning. The Development Cache sits behind all of it, so every build, every container, every pip install flows through one wall.

Start with the service that solves your next problem. Add the rest as the problem space grows.
