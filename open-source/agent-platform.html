§ 01 · Context Broker

Context Broker

Infinite conversation storage, presented through recipe-based context windows, enriched by durable facts and knowledge, curated through reinforcement learning, managed and metacoded through its own agent.

The Infinite Conversation

Every conversation — every message, every turn, every participant — is stored completely and indefinitely. This is the infinite conversation: not a log or an audit trail, but the medium through which agents think and accumulate knowledge over time.

Recipe Driven Context

A context window is not the conversation truncated to fit. It is a recipe — a purpose-built combination of context engineering techniques that assembles one agent's view of the conversation. We call these recipes build types. A build type can mix any form of compression, compaction, summarization, enrichment, RAG, CAG, or GraphRAG the engineer cares to assemble, to fit any token budget. Every agent gets its own optimized view of an infinite conversation — between one or many participants, human or agent — tuned to what that agent needs to reason well. This recipe-driven approach is the defining feature of the Broker.

Among the ingredients a recipe can draw on are two distinct memory layers. Episodic memory is the conversation record itself, progressively compressed as it ages. Semantic memory is a knowledge graph extracted from conversations in the background — entities, relationships, preferences, and decisions — answering what do I know about X rather than what was said about X. An engineering agent might be weighted toward semantic-dominant context; a persona agent maintaining continuity with a user, toward episodic; a strategy agent, toward injected reference material.

Assembly is proactive. Context is rebuilt for each participant in the background after every message, during idle time. When an agent reaches for its context, the assembled view is already waiting — no summarization latency at reasoning time. The complexity of compression, graph retrieval, and recipe evaluation is hidden from the agent entirely.

Context Engineering Agents

Two Context Engineering Agents ensure that optimal context is created, used, and improved — through contextual fact and knowledge extraction on one side, and direct short-cycle reinforcement learning on the other.

CEAs (server-side) ensures optimal compaction and extraction of durable facts and graph triples — not from isolated messages but from large chunks of conversation as part of the compaction process, so extraction happens in context rather than from isolated conversation turns. Because the CEAs is working within the context of a conversational chunk, it can make judgments about the quality of each fact as it is extracted, and store that quality alongside the fact — including potential expiration dates.

CEAc (client-side) runs at reasoning time as an agentic ReAct loop: decide what to search, search, rank, iterate if the first pass is thin, assemble the enrichment into the context window. And, more importantly, it feeds back to the Broker — closing the short-cycle reinforcement loop. Every retrieval is logged as a feedback event, feeding a four-dimension quality model: durability (how inherently stable this kind of fact is), confidence (how reliable the source was), usefulness (how often this fact has actually helped when retrieved), and an optional expiration gate. The final ranking multiplies relevance, trustworthiness, and memory quality — so the knowledge store gets sharper the more it is used, not noisier.

Imperator Managed Agentic Service

"Agentic service" means two things. They are both for agents, and each one has a dedicated agent — called an Imperator — that manages the service, making the service itself agentic.

Among the Imperator's many capabilities is its ability to metacode context window recipes. A person, or another agent, works with the Imperator directly to modify its own code because it knows best how its own system works.

Modular Hot Loading

Many aspects of the system can be modified and modularly hot-loaded — including build types and the Imperator itself. This means you can change the code without taking the service down.

Interfaces

  • MCP tool interface — ten-plus tools agents can call directly. Examples:
    • get_context — assemble and return an agent's context window
    • knowledge_search — search semantic memory and the knowledge graph
    • conv_search — search actual conversations across the entire store
  • OpenAI-compatible chat endpoint — the primary access path to the Imperator, provided via the industry-standard OpenAI-compatible endpoint protocol
  • Bundled chatbot UI — a lightweight web interface shipped with the Broker for talking to the Imperator directly

Stack

Python 3.12 · LangGraph + LangChain · FastAPI · PostgreSQL with pgvector · Mem0 · Neo4j · Infinity (embeddings) · Ollama (local inference) · Nginx.

§ 02 · eMAD Host

Plug-and-play agent orchestration.

eMAD stands for ephemeral Modular Agent Daemon. The Host is the runtime that loads them, routes to them, and manages their lifecycle. Install a new agent with pip install, and the Host picks it up — no restart, no monolithic deploy.

What it does

  • Dynamic agent loading — agents register via Python entry points; the Host discovers and routes to them
  • Routing by model name — any OpenAI client calling model: my-agent gets dispatched to the my-agent eMAD
  • Hot-swap — replace or upgrade an agent without taking the Host down
  • Built-in tools — sandboxed scripting, filesystem access, notifications, admin controls
  • Optional Context Broker integration — eMADs can offload persistent memory to a Context Broker for conversation continuity

Stack

Python 3.12 · LangGraph · FastAPI · PostgreSQL · Redis · Nginx.

§ 03 · Together

How they fit.

Context Broker and eMAD Host are independent but complementary.

  • Run Context Broker alone if you need durable conversational memory for an existing system.
  • Run eMAD Host alone if you need pluggable agent orchestration but don't need persistent memory.
  • Run both together — eMAD Host points at Context Broker, and every eMAD it hosts gets free durable memory across restarts.
