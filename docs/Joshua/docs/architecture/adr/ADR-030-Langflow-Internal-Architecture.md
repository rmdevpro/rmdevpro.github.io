# ADR-030: Langflow-Based Internal Architecture for Thought Engines

**Status:** Proposed
**Date:** 2025-12-20
**Deciders:** System Architect

## Context and Problem Statement

The current V0.7 implementation of a Multipurpose Agentic Duo's (MAD) Thought Engine is based on a "code-first" approach using LangChain's `AgentExecutor`. While functional, this approach presents challenges for rapid prototyping, iteration, debugging, and clear communication of the agent's internal logic. The reasoning process is defined in imperative Python code, which can be complex to build, modify, and visualize.

As we push the boundaries of agentic capabilities in a lab environment, we need an architecture that prioritizes development speed, modularity, and transparency without compromising the core principles of the Joshua ecosystem.

## Decision

We will adopt a new, "visual-first" internal architecture for MAD Thought Engines, based on **Langflow**. This decision refactors the internal DNA of a MAD into a hybrid model that separates the design-time environment from the run-time engine.

This architecture consists of three core pillars:

**1. Centralized Design-Time Environment (within `Hopper`):**
   *   The **`Hopper`** MAD will host a full, interactive Langflow UI instance as a development tool.
   *   This UI will serve as the centralized, visual "IDE" for designing, prototyping, and debugging the reasoning flows of any MAD in the ecosystem.
   *   `Hopper` will also provide a service to render these flows as static SVG/PNG images for documentation and review purposes.

**2. Declarative, Version-Controlled Artifact (`flow.json`):**
   *   The primary source of truth for a Thought Engine's reasoning *structure* will be a declarative `flow.json` file.
   *   This file, generated by `Hopper`'s Langflow UI, will be a version-controlled artifact within the target MAD's source code repository.
   *   Any custom programmatic logic required by the flow will be implemented in standard, testable Python files (`components/*.py`) that are version-controlled alongside the `flow.json`.

**3. Headless, Lightweight Run-Time Engine (`Joshua_Flow_Runner`):**
   *   MADs will **not** run a full Langflow instance. Instead, each MAD will use a new, shared library called `Joshua_Flow_Runner` for executing flows.
   *   This `Joshua_Flow_Runner` will be a lightweight, headless execution engine that programmatically loads and runs a MAD's `flow.json` by interacting with the `langflow` library's internal APIs.
   *   MAD servers use `Joshua_Flow_Runner` directly (part of Action Engine) - no wrapper needed.
   *   All library versions (`langflow`, `langchain`, etc.) will be strictly pinned to ensure a stable and reproducible runtime environment.

## Consequences

*   **Positive:**
    *   **Increased Development Velocity:** Visual design allows for significantly faster prototyping and iteration of complex agentic logic.
    *   **Enhanced Transparency & Communication:** The visual representation of the `flow.json` provides a clear, "read-only" interface for understanding and debugging an agent's internal thought process, benefiting both developers and non-technical stakeholders.
    *   **Improved Modularity:** Promotes the creation of reusable visual components and sub-flows.
    *   **Low Runtime Overhead:** By separating the design-time UI from the run-time, individual MADs remain lightweight, using only a headless Python runner.
    *   **Architectural Purity:** This change has **zero impact** on inter-MAD communication. The `Joshua_Communicator` library's role is unaffected, and the TE/AE client/server relationship via MCP remains the same, preserving the core architectural principles of the ecosystem.

*   **Neutral:**
    *   **New Library Development:** One new shared library is required:
        - `Joshua_Flow_Runner`: Headless Langflow execution engine (medium effort, requires integration with Langflow internal APIs). MAD servers use this directly.
    *   **Architectural Evolution:** The `joshua_thought_engine` evolves from a library with implementation code to a pure dependency aggregator that declares dependencies on PCP component libraries (joshua_dtr, joshua_imperator, joshua_cet, etc.). See ADR-031 and ADR-032 for details.

*   **Negative:**
    *   **Dependency on Internal APIs:** The `Joshua_Flow_Runner` will be tightly coupled to the internal, non-public APIs of the `langflow` library. While version pinning mitigates unexpected breakage, planned upgrades to `langflow` will require a corresponding maintenance and re-validation effort for the `Joshua_Flow_Runner`.
    *   **Shift in Testing Paradigm:** Unit testing the logic within a `flow.json` is less direct than testing a Python class. The testing approach will need to adapt to focus more on integration-style tests of the flow's inputs and outputs.
