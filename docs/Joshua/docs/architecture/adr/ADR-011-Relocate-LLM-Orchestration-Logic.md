# ADR-011: Relocate LLM Orchestration Logic to the Calling MAD's PCP

**Status:** Accepted (Superseded by ADR-021/ADR-023/ADR-035)

**Context:**
As Fiedler's role has been refined (ADR-008, ADR-009), a key architectural question has emerged: who is responsible for orchestrating complex LLM usage patterns, such as fallbacks, multi-model synthesis, or critique loops? An initial thought was to centralize this "smart" orchestration within Fiedler itself. However, this would make Fiedler a cognitive bottleneck and violate Joshua's principle of distributed intelligence, where each MAD is an autonomous agent. It would also create a redundant cognitive layer, as the calling MAD's own Progressive Cognitive Pipeline (PCP) is already designed to handle complex, reasoned orchestration.

**Decision:**
The responsibility for all **application-level LLM orchestration** will be formally relocated from Fiedler to the **calling MAD's own Thought Engine (PCP)**. Fiedler's role is strictly limited to **transport-level resilience** and providing expert LLM recommendations.

1.  **Calling MAD's Responsibility (Application-Level Orchestration):**
    *   The calling MAD's PCP (specifically its Imperator for novel tasks or its LPPM for learned workflows) is responsible for defining and executing the "how" of LLM usage.
    *   This includes patterns like:
        *   **Strategic Fallbacks:** Deciding to call an alternative model if the primary choice fails to produce a satisfactory result.
        *   **Multi-Model Synthesis:** Orchestrating complex workflows like a `(Draft -> Critique -> Synthesize)` loop that involves multiple calls to Fiedler.
        *   **Tool Chaining:** Integrating LLM calls with other tools from its own or other MADs' Action Engines.
    *   **Implementation:** These orchestrations will be implemented within the MAD's Thought Engine using frameworks like LangChain Expression Language (LCEL) or LangGraph, as dynamically generated by the Imperator or as compiled in a learned LPPM process.

2.  **Fiedler's Responsibility (Transport-Level Resilience & Expertise):**
    *   Fiedler is responsible for ensuring that a single, requested LLM call succeeds if possible. Its Action Engine will handle:
        *   **Automatic Retries:** Automatically retry on transient network or API errors (e.g., `429 Too Many Requests`, `503 Service Unavailable`) using an exponential backoff strategy. This is invisible to the calling MAD.
        *   **Intelligent Error Reporting:** If a provider is terminally unavailable after multiple retries, Fiedler will not return a raw network exception. It will return a structured error message that includes an **intelligent recommendation** for a suitable alternative model, based on its expertise of the LLM landscape.
    *   Fiedler's cognitive role remains that of a **recommender** (`fiedler_recommend_model`), not an orchestrator of other MADs' workflows.

**Consequences:**

*   **Positive:**
    *   **Reinforces Distributed Cognition:** This decision firmly places cognitive and orchestration responsibility with the agent performing the task, adhering to Joshua's core architectural principles.
    *   **Avoids a Central Bottleneck:** Prevents Fiedler from becoming a complex, single point of failure for all LLM-related workflows.
    *   **Empowers MADs:** Makes individual MADs smarter and more autonomous. They own their reasoning processes and can develop unique, domain-specific ways of using LLMs.
    *   **Leverages the PCP:** Correctly utilizes the Imperator and LPPM as the intended layers for reasoned orchestration, avoiding architectural redundancy.
    *   **Clear Separation of Concerns:** Creates a clean and robust boundary. Fiedler worries about the *delivery* of a single LLM call; the calling MAD worries about the *strategy* of how to use one or more calls.

*   **Negative:**
    *   **Increased MAD Complexity:** The cognitive burden on each MAD's Thought Engine is higher, as it must now define its own LLM usage patterns instead of calling a single, high-level orchestration tool. This is an accepted and desired complexity.

*   **Neutral:**
    *   This decision provides a definitive clarification of the division of responsibility between Fiedler and all other MADs, ensuring a consistent and scalable architecture for LLM usage across the ecosystem.

## Appendix A: Detailed Architectural Rationale

### A.1. The "Smart Agent, Not Smart Pipe" Principle

The core driver of this decision is the principle that intelligence should reside in the autonomous agents (the MADs), not in the communication pipes between them. My previous line of thinking was heading toward making Fiedler an overly "smart pipe"—a centralized orchestrator that would dictate *how* other MADs should think.

This is antithetical to Joshua's design. The correct model is that each MAD's PCP is its own brain. Hopper, as a "Builder," should have its own unique, learned patterns for how it uses LLMs to write and refactor code. McNamara, as a "Security Analyst," should have entirely different patterns for how it uses LLMs to analyze threats. Centralizing this logic in Fiedler would destroy that specialized, distributed intelligence.

### A.2. Analogy: The Browser vs. The User

A powerful analogy helped clarify the separation of concerns:

*   **Fiedler is the Web Browser:** It is a sophisticated tool for accessing a service (the web/LLMs). It handles low-level resilience for you—if a TCP packet drops, it retries; if a server is slow, it waits. If a website is completely down, it gives you a clear, actionable error page ("Server Not Found"), not a raw network error. It is an expert on the transport layer.
*   **The Calling MAD is the User:** The user decides their own high-level workflow. "First, I'll search on Google. Then, I'll open the top three links in new tabs. I'll read each one and synthesize a summary." The browser doesn't decide this workflow for you; you direct the browser.

This ADR formalizes that Fiedler is the "browser" for LLMs, and each MAD is the "user" directing its own research strategy.

### A.3. How This Enables Downward Knowledge Distillation

This decision is critical for the PCP's learning mechanism. By relocating LLM orchestration logic to the calling MAD's PCP:

1.  **Imperator (Tier 4) Innovates:** Initially, a MAD's Imperator will have to reason from scratch to create a complex LLM orchestration (e.g., a "Draft -> Critique -> Synthesize" LangGraph). This is slow and expensive.
2.  **LPPM (Tier 2) Automates:** After observing the Imperator successfully use this graph pattern multiple times to solve a recurring problem, the LPPM will learn it. The next time that problem occurs, the LPPM can now execute that same LangGraph automatically, without involving the Imperator.
3.  **Action Engine Encapsulates:** Eventually, a very common and stable orchestration pattern (like a resilient call with fallbacks) can be encapsulated as a "Composite Tool" in the Action Engine, making it even easier for the PCP to use.

If this orchestration logic were hidden inside Fiedler, the calling MAD's PCP would never see it and would never be able to learn from it or automate it. Placing the logic in the calling MAD is the **only way** to enable this crucial "downward knowledge distillation."
