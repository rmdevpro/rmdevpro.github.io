# Joshua System Evolutionary Roadmap

**Version:** 3.0 - Unified
**Status:** Authoritative
**Last Updated:** 2025-11-30
**Alignment:** ADR-020 (System Evolution Roadmap)
**Purpose:** To define the strategic, phased evolution of the Joshua ecosystem from foundational maturity through cognitive pipeline rollout, eMAD revolution, and strategic autonomy activation.

---

## Guiding Principle: Capabilities Unlock New Versions

Joshua's evolution follows a multi-faceted approach where the introduction of new system-level capabilities enables corresponding upgrades in MAD-level versions. This roadmap provides a clean separation of concerns, progressing from foundational infrastructure and core cognitive layers to advanced scalability and strategic leadership.

**Key Concept:** This roadmap defines the project's authoritative timeline, clarifying the progression of the *system* through named phases and the corresponding target version of the *MADs* at the completion of each phase. There are no "System-Level Versions" like "Joshua is at V6.0"; the correct terminology is, "The Joshua ecosystem is in Phase 4 (eMADs), and the core MADs are being upgraded to the V6.0 standard."

### Version Progression Summary

**V0.1 (Basic)** - Action Engine only, deterministic tools, pure Joshua-native

**V0.5 (Reasoning)** - Thought Engine introduced (custom Python Imperator), basic LLM-based reasoning

**V0.7 (Flow-Based)** - Flow-based architecture (LangFlow), logic in declarative flows, Imperator without RAG

**V0.8 (Context-Aware Reasoning)** - Henson MAD built with Qdrant RAG, Imperator library updated to integrate context retrieval

**V0.10 (Federation Ready)** - Hardened, tested, quality-gated, flawless Direct Communication

**V1.0 (Bus Migration)** - Re-platformed to Rogers Conversation Bus (pure Kafka), Joshua_Communicator with native Kafka

**V2.0 (DTR - Reflexive Routing)** - PCP Tier 1, Deterministic Task Router, fast-path routing

**V3.0 (LPPM - Learned Workflows)** - PCP Tier 2, Learned Process and Procedure Memory, CrewAI + LangGraph

**V4.0 (CET - Context Engineering)** - PCP Tier 3, Context Engineering Transformer with LoRA adapters

**V5.0 (CRS - Metacognitive Validation)** - PCP Tier 5, Cognitive Response System, pattern-matching quality gate

**V6.0 (eMAD-Aware)** - Can delegate to eMAD teams via Kaiser, horizontal scalability

**V7.0 (Joshua-Aware)** - Ultimate maturity, can receive strategic directives from Joshua MAD, constitutional guidance

## The 6-Phase Evolution

### Phase 1: Foundation (MAD Version Target: V0.10)

**Status:** IN PROGRESS

**Primary Goal:** Build and mature the Core Fleet of persistent MADs to a "Federation Ready" state, capable of V0 Direct Communication.

**Key Activities & Architectural Changes:**
- All core MADs refactored/built to **V0.7 standard** (ADR-032: Flow-Based Architecture)
- Sutherland (GPU orchestrator) matured from V0.1 to **V0.7 federated "Local Warden" model** (ADR-015)
- Hamilton (Monitoring) built to **V0.7 level**
- Deming (CI/CD, Quality Gate) built to **V0.7 level**
- Henson (Context) built to **V0.8 level**, providing Qdrant RAG services (ADR-026, ADR-039)
- **Malory (Web Browser Automation) built to V0.7 standard** - Flow-based with Node.js/Playwright subprocess nodes (ADR-024 Non-Python Subsystem Pattern, ADR-032 Flow-Based Architecture)
- All MADs hardened, tested, and quality-gated for flawless Direct Communication

**Core Fleet MADs (Phase 1):**
1. Hopper - Autonomous Software Engineering
2. Turing - Secrets & Credentials Management
3. Starret - Git Management
4. Fiedler - AI Model Ecosystem Orchestrator
5. Deming - CI/CD, Quality & Test Management
6. Malory - Web Browser Automation & UI Testing
7. Grace - Human Interface
8. Sutherland - GPU Management
9. Henson - Context Engineering (Fast RAG)
10. Hamilton - System Monitoring & Observability

**Version Progression (V0 Path):**
- **V0.1 (Basic):** Action Engine only, deterministic tools, pure Joshua-native
- **V0.5 (Reasoning):** Thought Engine introduced (custom Python Imperator), basic LLM-based reasoning
- **V0.7 (Flow-Based):** Flow-based architecture (LangFlow), logic in declarative flows, Imperator without RAG
- **V0.8 (Context-Aware Reasoning):** Henson MAD built with Qdrant RAG, Imperator library updated to integrate context retrieval
- **V0.10 (Federation Ready):** Hardened, tested, quality-gated, flawless Direct Communication

### Phase 2: Conversation (MAD Version Target: V1.0)

**Status:** PLANNED

**Primary Goal:** Establish the durable, learnable communication substrate for the entire ecosystem through migration from the legacy websocket-based MCP network to the Rogers Conversation Bus.

**Key Activities & Architectural Changes:**
- Re-platform the entire V0.10-certified Core Fleet to communicate exclusively over the **Rogers Conversation Bus** (ADR-007: Pure Kafka Architecture v2)
- MADs' Joshua_Communicator library updated to handle communication via Rogers (native Kafka producer/consumer)
- Babbage configured to consume the Kafka log for permanent conversation archival and CQRS Read Model construction
- Pure Kafka architecture replaces websocket MCP for all inter-MAD communication
- Rogers acts as control plane manager (topic lifecycle, service registry, administrative operations)
- Sam MAD provides synchronous-to-asynchronous bridge for external MCP clients (e.g., Claude Code)

**Phase 2 MADs:**
11. Rogers - Conversation Bus Controller (Control Plane)
12. Sam - Communication Bus Gateway (Sync-to-Async Bridge)
13. Babbage - Semi-Structured Data Management (CQRS Read Models)
14. Horace - File System Management

**V1.0 Definition (Bus Migration):**
- Re-platformed to Rogers Conversation Bus (pure Kafka)
- Uses Joshua_Communicator with native Kafka implementation instead of joshua_network
- Asynchronous, event-driven communication
- All conversation data flows through durable Kafka log
- "Durable Fast Lane" architecture: OS page cache for low-latency + disk persistence

### Phase 3: Cognition (MAD Version Target: V5.0)

**Status:** PLANNED

**Primary Goal:** Roll out the full Progressive Cognitive Pipeline (PCP) to the Core Fleet, achieving complete cognitive capabilities through four major upgrades.

**Key Activities & Architectural Changes:**

**V2.0: DTR (Tier 1 - Reflexive Routing)**
- DTR added to every MAD's Thought Engine (ADR-006: Learned, Content-Based Router)
- Fast-path routing before expensive LLM processing
- Handles 60-80% of deterministic traffic in microseconds

**V3.0: LPPM (Tier 2 - Learned Workflows)**
- LPPM added to every MAD's Thought Engine (ADR-004, ADR-005: PCP Cognitive Flow with DTR Routing)
- Multi-step workflow execution using CrewAI concepts and LangGraph
- Handles 15-30% of learned patterns in milliseconds

**V4.0: CET (Tier 3 - Context Engineering)**
- Henson upgraded to host full **CET transformer model** (ADR-003, ADR-012: Henson as Context Engineer)
- All MADs upgraded to utilize advanced context assembly services
- Replaces Fast RAG with learned context engineering transformer
- LoRA adapters for personalized context assembly

**V5.0: CRS (Tier 5 - Metacognitive Validation)**
- CRS added to every MAD's Thought Engine (ADR-001: Pattern-Matching and Anomaly Detection)
- Pattern-matching and anomaly detection for quality gate
- Metacognitive validation and self-correction
- Full PCP operational: **DTR → LPPM → CET → Imperator (Tier 4) → CRS**

### Phase 4: eMADs (MAD Version Target: V6.0)

**Status:** PLANNED

**Primary Goal:** Introduce the infrastructure for ephemeral, scalable intelligence and upgrade all Core MADs to utilize it.

**Key Activities & Architectural Changes:**
- **Introduce Kaiser:** The Kaiser MAD is built (ADR-017: eMAD Lifecycle Management) as the team composition expert
- **Introduce Moses:** The Moses MAD is built (ADR-018: Federated Container Placement Orchestrator) running on every host
- **Upgrade Core MADs to be "eMAD-aware":** All Core Fleet MADs upgraded to **V6.0**, gaining the capability to delegate tasks to eMAD teams via Kaiser (ADR-009: Evolution from "Dumb LLM Calls" to "Intelligent eMAD Teams")
- MAD Thought Engines updated to reason about eMAD delegation
- MAD Action Engines updated to call Kaiser for team composition
- Horizontal scalability through ephemeral intelligence

**Phase 4 MADs:**
15. Kaiser - eMAD Lifecycle Management (Team Composition Expert)
16. Moses - Federated Container Placement Orchestrator

**V6.0 Definition (eMAD-Aware):**
- Thought Engine can delegate to eMAD teams via Kaiser
- Evolves from using single AI model access nodes to orchestrating eMAD teams for complex tasks
- Horizontal scalability enabled through ephemeral MADs
- Persistent MADs orchestrate ephemeral teams for complex tasks

**Supporting Infrastructure - LangFlow to LangGraph Compiler:**
- Development after Phase 4 (eMADs)
- Converts LangFlow flows to LangGraph for 5x production performance improvement (30s vs 148s measured)
- Maintains visual development workflow (LangFlow UI + Selenium for Hopper)
- **References:** ADR-046

- Top-down orchestration operational across entire ecosystem

## Progressive Cognitive Pipeline (PCP) Architecture

The PCP is the cognitive backbone of the Joshua system, rolled out progressively through Phases 3-5:

**Tier 1: DTR (V2.0)** - Microseconds
- **Deterministic Task Router**
- Learned, content-based routing
- Fast-path for 60-80% of deterministic traffic
- Pattern matching with sub-millisecond latency

**Tier 2: LPPM (V3.0)** - Milliseconds
- **Learned Process and Procedure Memory**
- Pattern-based workflow execution
- Handles 15-30% of learned workflow patterns
- CrewAI concepts + LangGraph execution

**Tier 3: CET (V4.0)** - Sub-Second
- **Context Engineering Transformer**
- Advanced, learned context assembly
- LoRA adapters for personalized context
- Served by Henson MAD

**Tier 4: Imperator (V0.5+)** - Seconds
- **Core LLM Reasoning Engine**
- Full deliberative processing
- Handles 5-10% of novel, complex reasoning
- Multiple LLM backend support via Fiedler

**Tier 5: CRS (V5.0)** - Sub-Second
- **Cognitive Response System**
- Pattern-matching and anomaly detection
- Quality gate and metacognitive validation
- Self-correction and confidence scoring

**Complete PCP Flow:**
```
Request → DTR (fast path?) → LPPM (learned pattern?) → CET (context assembly)
       → Imperator (LLM reasoning) → CRS (validation) → Response
```

## Development Philosophy

**MAD = Multipurpose Agentic Duo**

Each MAD consists of two engines working as a duo:
- **Thought Engine**: The cognitive system that understands, reasons, and makes decisions
- **Action Engine**: The execution system that performs tasks and manages domain-specific tools

**Communication Model:**
- MADs communicate through free-form conversations over the Rogers Conversation Bus
- Conversations can contain deterministic commands, fixed data structures, or natural language prose
- Direct Dispatcher for JSON-RPC 2.0 formatted calls (programmatic access)
- LLM Fallback for conversational prompts (natural language interface)

**Learning & Evolution:**
- System learns and self-improves over time through the Progressive Cognitive Pipeline
- DTR learns routing patterns, LPPM learns workflows, CET learns context assembly
- CRS provides metacognitive validation and quality gates
- Babbage CQRS Read Models provide training data for all learning tiers
