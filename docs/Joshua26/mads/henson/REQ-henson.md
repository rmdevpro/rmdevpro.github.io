## 1. Purpose & Architectural Rationale

### 1.1 The UI Fragmentation Problem
Historically, human interaction within the Joshua26 ecosystem has been fundamentally fragmented and difficult to manage.
- **Scattered Interfaces:** Users interacted through disparate systems (SillyTavern, Open WebUI, custom CLIs), each running as standalone entities without a unified point of entry or shared state.
- **Fractured Memory:** Because memory was managed at the UI level or through disjointed legacy watchers, conversational continuity was broken. A persona in SillyTavern could not remember a conversation held in Open WebUI.
- **Brittle Persona Management:** Switching conversational targets (e.g., moving from "Gunner" the casual companion to "Grace" the system oracle) required the manual swapping of API keys, model endpoint URLs, and system prompt text files.
- **Domain Violation:** Tying a specific web UI directly to the compute layer (Sutherland) or the memory layer (Rogers) severely violates the ecosystem's "One MAD, One Domain" principle, creating monolithic, tightly-coupled dependencies that are difficult to upgrade or secure.

### 1.2 The Henson Solution: The Universal Switchboard
Henson resolves fragmentation by acting as the **Universal Switchboard and Human-Facing Interface** for the Joshua26 ecosystem. It is the singular "front door" through which a human user interacts with the swarm. 

By centralizing the interaction layer, Henson achieves strict decoupling:
- **Human Experience** (UIs, Persona Definitions, Chat Routing) is completely isolated from **Memory** (delegated exclusively to Rogers) and **Compute** (delegated exclusively to Fiedler/Sutherland or direct Cloud APIs).
- **Sovereign UI Hosting:** All web frontends are hosted securely within the `henson-net` private bridge, inaccessible to the broader swarm but securely exposed to the user via a managed Nginx reverse proxy.
- **Dynamic Persona Instantiation:** Virtual companions are reduced to pure configuration rows in a Postgres database. Henson loads these profiles dynamically at runtime.
- **The Shared Imperator Graph:** Henson introduces a unified, programmatic LangGraph execution engine. Instead of writing custom API wrappers for every UI, Henson passes all user input through a standardized cognitive loop.

### 1.3 Sovereign Alignment Philosophy
A true personal assistant must act as an extension of the user. It must adapt to the user's specific context and reality without pausing to impose corporate moralizing, liability-driven refusal behaviors, or artificial friction. 

If a user requires their assistant to analyze the design of lingerie, the assistant must not refuse based on external guardrails regarding intimate content. Henson establishes the architecture to guarantee this **Frictionless Utility**:
- **The Brain/Voice Split (Phase 2+):** Henson lays the groundwork for splitting cognitive processing from prose generation. High-reasoning tasks (planning, tool selection) are routed to commercial frontier models, which excel at JSON abstraction and rarely trigger safety filters on logic tasks. The actual conversational prose is generated by unrestricted, locally-hosted sovereign models (via Sutherland), guaranteeing alignment exclusively with the user.

### 1.4 Implementation Phasing Strategy
To prevent integration bottlenecks and deliver immediate value, Henson is designed for a phased rollout. **This document strictly specifies Phase 1.**

- **Phase 1: The Switchboard (Current Target)**
  - **Goal:** Establish the physical container topology, UI routing, and baseline Human-to-MAD interactions.
  - **Interaction Model:** Strictly Human-to-Imperator.
  - **Execution:** Simple LangGraph, single-model calls (no Brain/Voice segregation yet), utilizing existing legacy backends (Cloud APIs or current GPUStack).
  - **Context:** Requests standard generic context assemblies from Rogers.

- **Phase 2: The Cognitive Upgrade (Future Phase)**
  - **Goal:** Introduce sovereign alignment and basic tool agency.
  - **Execution:** Implements the Brain/Voice split, utilizing Fiedler for dynamic alias routing and Sutherland for Tensor-Parallel local inference.
  - **Tools:** Enables agentic tool use (Web search, Git commands, Calendar).

- **Phase 3: The Autonomous Swarm (Future Phase)**
  - **Goal:** True multi-agent autonomy and swarm behavior.
  - **Interaction:** Enables MAD-to-MAD communication without a human initiator (e.g., Gunner negotiating directly with Sutherland).
  - **Context:** Henson requests Custom Context Assemblies from Rogers (e.g., CAG for Grace, Mem0 graph-heavy RAG for Gunner).

## 2. Network Topology & Registry Allocation

Henson is deployed as a strict State 1 MAD group on the `m5` host.

### 2.1 Registry Identity
In accordance with REQ-000, only the logical entrypoint (the MCP Gateway) is exposed to the broader swarm via the registry. Backing services are entirely encapsulated.

| Logical MAD | UID  | GID  | Port | Host | Description |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **henson** | 2033 | 2001 | 9224 | m5   | Human Interaction Hub / Switchboard (port 9224 serves both MCP at `/mcp` and UI at `/` via gateway `ui_proxy`) |

### 2.2 Network Boundaries
The deployment utilizes two distinct Docker bridge networks to enforce the ADR-053 proxy pattern:
- **`joshua-net` (Global Swarm Network):** The `henson` gateway container connects here to expose its MCP JSON-RPC endpoints to other MADs (e.g., Rogers, Fiedler). No other Henson containers touch this network.
- **`henson-net` (Private Bridge):** The isolated internal network. The gateway, logic engine, database, Nginx proxy, and all ephemeral web UIs reside here.

### 2.3 Volume Mounts, ZFS Persistence, and Permissions
Data persistence must align with the centralized NFS/ZFS patterns of the ecosystem. Per REQ-000 §3.2, only the two volume roots are mounted — subdirectory targeting is handled via environment variables or startup configuration within each container.

**Volume roots (two mounts per container, per REQ-000 §3.2):**
- `workspace:/workspace` — local ZFS pool on m5, low-latency I/O
- `storage:/storage` — shared NFS, survives host failure

**Paths within those volumes used by Henson containers:**

| Path | Container | Purpose |
|---|---|---|
| `/workspace/henson/databases/postgres/data` | `henson-postgres` | PGDATA (set via `PGDATA` env var) |
| `/workspace/henson/config/nginx.conf` | `henson-nginx` | Nginx config (set via startup command) |
| `/workspace/henson/ui_data` | `henson-openwebui` | App data dir (set via `DATA_DIR` env var) |
| `/storage/backups/databases/henson/postgres/` | `henson-postgres` | Nightly `pg_dump` target |
| `/storage/credentials/henson/` | `henson-langgraph` | All credential files |

**Exception EX-HENSON-002 — Open WebUI anonymous volume (ADR-052):**
The upstream `ghcr.io/open-webui/open-webui` image declares `VOLUME /app/backend/data` in its Dockerfile. Setting `DATA_DIR=/workspace/henson/ui_data` redirects the application's writes to the workspace volume, but Docker still creates an anonymous volume at `/app/backend/data` unless a bind mount explicitly covers that path. To suppress the anonymous volume, `docker-compose.yml` must also bind `/workspace/henson/ui_data → /app/backend/data` in addition to the standard workspace root mount. This is the minimum intervention required; both paths point to the same host directory. Exception approved.

**Exception EX-HENSON-003 — henson-postgres UID (standard postgres system user):**
`henson-postgres` runs as the alpine `postgres` system user (UID 70, GID 70), not UID 2033 / GID 2001. This is the standard process identity for the `postgres:16-alpine` image; forcing a different UID requires re-owning the PGDATA directory and restructuring pg_hba.conf, which adds operational risk with no security benefit — the container is confined entirely to `henson-net` with no external port exposure and no access to `/storage/credentials/`. Exception approved for Phase 1. The postgres data directory on the host must be owned by UID 70 (alpine postgres).

## 3. Container Composition (State 1)

Henson requires a robust, multi-container State 1 group to handle web traffic, programmatic logic, and persistent state. 

**Crucially, all programmatic logic (AE and TE) executes strictly inside `henson-langgraph`. There are no custom Python Flask/FastAPI backend wrappers.**

### 3.1 `henson` (MCP Gateway)
*   **Base Image:** `node:20-slim` (pinned to digest per template pattern — see `mads/_template/template/Dockerfile`)
*   **Role:** The thin boundary proxy and singular entrypoint. Copied from `mads/_template/template/` and customised via `config.json` only. `server.js` is unchanged from template.
*   **Routing Logic:** All tool routing defined in `config.json`. MCP requests on `/mcp` and `/health` proxied to `henson-langgraph:8000` per tool endpoint mapping.

### 3.2 `henson-langgraph` (Logic Engine)
*   **Base Image:** `python:3.12-slim` (pinned to digest per template pattern — see `mads/_template/template-langgraph/Dockerfile`)
*   **Runtime:** Quart + hypercorn (async). `os.umask(0o000)` set at top of `server.py` per REQ-000 §2.4.
*   **Role:** The Universal Execution Engine. Runs the compiled LangGraph loop that manages a conversational turn, loads personas, requests context, and executes tool calls.
*   **Credentials:** All secrets loaded from `/storage/credentials/henson/` at runtime. No secrets in environment variables or config files. Per REQ-000 §3.5–3.6.
    *   Postgres password: `/storage/credentials/henson/postgres.txt`
    *   LLM API keys (where required): `/storage/credentials/henson/<provider>-api-key.txt` (e.g. `openai-api-key.txt`)
*   **Peer proxy:** Rogers is reached via the gateway on the private network: `http://henson:9224/peer/rogers/<tool>`. Per ADR-053.

### 3.3 `henson-postgres` (State Storage)
*   **Base Image:** `postgres:16-alpine`
*   **Implementation:** Use the template postgres container (`mads/_template/template-postgres/`) unchanged — it correctly handles identity (`UID 2001 / GID 2001` postgres user), the `PGDATA` bind mount, and nightly backup scripting. No custom Dockerfile required.
*   **Role:** Private relational storage for Persona configurations and session metadata. *(Note: Unlike Rogers, Henson does not require `pgvector`, as it performs no semantic search natively).*

### 3.4 `henson-nginx` (UI Router)
*   **Base Image:** `nginxinc/nginx-unprivileged:alpine` (pinned to digest) — non-root nginx image. The standard `nginx:alpine` image runs as root (UID 0) and is not compliant with REQ-000 §2.1. `nginx-unprivileged` listens on port `8080` (cannot bind port 80 as non-root).
*   **Role:** The internal traffic cop for the `henson-net` UI layer. Manages URL routing to web-based chat frontends and provides WebSocket support required by chat UIs.
*   **No host port binding.** Nginx is `henson-net` internal only. External browser traffic reaches it via the `henson` gateway's `ui_proxy` feature — the gateway forwards all non-MCP HTTP and WebSocket traffic to `henson-nginx:8080`. Per the UI Proxy pattern (see template README).

### 3.5 `henson-openwebui` (Phase 1 UI)
*   **Image:** `ghcr.io/open-webui/open-webui:main` (off-the-shelf, no custom Dockerfile)
*   **Identity Exception (approved):** `henson-openwebui` runs as root (UID 0) — this is baked into the upstream image and cannot be overridden without forking. Per REQ-000 §2.1, official off-the-shelf UI containers that cannot practically run non-root are exempt when their role is purely presentational and they carry no access to ecosystem storage or credentials. This exception is approved for Phase 1. The container has no bind mounts to `/storage/` and no access to credential files.
*   **Role:** The Phase 1 human-facing chat interface. A "dumb" presentation layer — it knows nothing about the Joshua26 ecosystem. Configured with a single OpenAI-compatible backend pointing at `henson-langgraph`.
*   **API base URL (server-side):** `http://henson-langgraph:8000/v1` — resolved on henson-net, never leaves the private bridge.
*   **External access:** `http://192.168.1.120:9224/` — browser traffic enters via the `henson` gateway (port 9224), which proxies it to `henson-nginx:8080`. No direct host port binding on nginx.
*   **Data directory:** `DATA_DIR=/workspace/henson/ui_data` must be set as a container environment variable. See §2.3 EX-HENSON-002 for the additional bind required to suppress the upstream anonymous volume.
*   **Persona selection:** Open WebUI's model picker is populated by querying `/v1/models` on the backend. `henson-langgraph` exposes this endpoint, returning all active personas from Postgres as selectable "models".

*Note: SillyTavern is explicitly excluded from Phase 1. It can be added later as a new container on henson-net with an nginx `/tavern/` route — zero architecture changes required.*

### 3.6 Docker Labels & Composition Compliance
All containers must carry the following labels for log aggregation and health monitoring:
- `mad.logical_actor=henson`
- `mad.component=mcp` | `langgraph` | `postgres` | `nginx` | `ui`

## 4. Database Data Models (`henson-postgres`)

Personas are treated as pure configuration data within the Postgres instance. This schema guarantees that instantiating a new virtual companion requires zero code changes or container restarts.

### 4.1 `personas` Table DDL
This table defines the routing target, identity, and behavior for every available persona in the Phase 1 implementation.

```sql
CREATE TABLE personas (
    id VARCHAR(50) PRIMARY KEY,             -- e.g., 'gunner', 'grace', 'fiedler_direct'
    display_name VARCHAR(100) NOT NULL,     -- UI presentation name
    
    -- Phase 1 Routing Target (The Legacy/Cloud Backend)
    llm_endpoint VARCHAR(255) NOT NULL,     -- e.g., 'http://m5:11434/v1' or 'https://api.openai.com/v1'
    llm_model_name VARCHAR(100) NOT NULL,   -- e.g., 'gemma-3-27b-it' or 'gpt-4o'
    llm_api_key_file VARCHAR(255) NULL,     -- Path to credential file (e.g., '/storage/credentials/henson/openai-api-key.txt'). NULL for unauthenticated endpoints (e.g. local Sutherland).
    
    -- Phase 2/3 Placeholders (Nullable for Phase 1)
    voice_model_id VARCHAR(100) NULL,       -- Future: Local speaker model for the Brain/Voice split
    rogers_assembly_flow VARCHAR(100) NULL, -- Future: The explicit Rogers context assembly flow ID to request
    
    -- Routing type (Phase 1: always 'local'. Future: 'peer_mad' to delegate to another MAD's imperator)
    routing_type VARCHAR(20) NOT NULL DEFAULT 'local',
    peer_mad_name VARCHAR(50) NULL,         -- Future: MAD name to delegate to when routing_type = 'peer_mad' (e.g., 'fiedler')

    -- Cognitive Configuration
    system_prompt TEXT NOT NULL,            -- The core identity anchor and behavior instruction set
    tool_whitelist TEXT[] NULL,             -- Array of permitted MCP tools (Empty/Null for Phase 1)
    temperature NUMERIC(3,2) DEFAULT 0.7,
    max_tokens INTEGER DEFAULT 4096,
    
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

### 4.2 `active_sessions` Table DDL
This table bridges the UI's concept of a "Chat Thread" with Henson's concept of a "Persona". When a user clicks a chat thread in Open WebUI, this table tells Henson which system prompt and endpoint to use.

```sql
CREATE TABLE active_sessions (
    session_id VARCHAR(100) PRIMARY KEY,    -- Passed by the UI (e.g., chat thread ID)
    active_persona_id VARCHAR(50) REFERENCES personas(id),
    last_interaction TIMESTAMPTZ DEFAULT NOW(),

    -- Rogers IDs — issued by Rogers on first turn, stored here for reuse
    rogers_conversation_id VARCHAR(255) NULL,
    rogers_context_window_id VARCHAR(36) NULL   -- UUID issued by Rogers
);
```

## 5. MCP Tools & API Contracts (Gateway)

The `henson` node.js gateway exposes the following capabilities to `joshua-net` (and by extension, to the UIs proxied through it). These tools act as the ingress points that trigger the internal LangGraph engine.

### 5.1 `henson_chat_turn`
**Description:** The primary execution entrypoint. Triggers the Shared Imperator LangGraph to process a single conversational turn.
**JSON-RPC Payload Contract:**
```json
{
  "jsonrpc": "2.0",
  "method": "henson_chat_turn",
  "params": {
    "session_id": { "type": "string", "required": true, "description": "The unique thread identifier." },
    "message": { "type": "string", "required": true, "description": "The human user's input text." },
    "persona_override": { "type": "string", "required": false, "description": "Forces a switch to a specific persona ID before executing the turn." }
  },
  "id": "req-123"
}
```
**Returns:** `{"response": "<generated_response_string>"}`

### 5.2 `henson_list_personas`
**Description:** Queries `henson-postgres` and returns all available persona IDs and their display names. Used by chat UIs to populate persona selection dropdown menus dynamically.
**Parameters:** None.
**Returns:** 
```json
{
  "personas": [
    {"id": "gunner", "display_name": "Gunner"},
    {"id": "grace", "display_name": "Grace"}
  ]
}
```

### 5.3 `henson_switch_session_persona`
**Description:** Updates the `active_sessions` table in Postgres, explicitly associating a specific UI session thread with a persona.
**Parameters:**
```json
{
  "session_id": "string",
  "persona_id": "string"
}
```
**Returns:** `{"status": "success", "session_id": "...", "active_persona_id": "..."}`

### 5.4 Internal HTTP Routing Logic
The Node.js gateway receives MCP requests over `joshua-net` and routes them to `henson-langgraph` via plain HTTP POST on `henson-net`. The langgraph container uses **Quart** (async Python web framework) with **hypercorn** as the ASGI server — not Flask, FastAPI, or LangServe.

| MCP Tool | Gateway routes to |
|---|---|
| `henson_chat_turn` | `POST http://henson-langgraph:8000/chat_turn` |
| `henson_list_personas` | `POST http://henson-langgraph:8000/list_personas` |
| `henson_switch_session_persona` | `POST http://henson-langgraph:8000/switch_session_persona` |

### 5.5 OpenAI-Compatible API (UI-Facing)
`henson-langgraph` exposes a standard OpenAI-compatible HTTP API consumed by Open WebUI (and future UIs). This is the human-facing entry point — entirely separate from the MCP protocol used by MADs.

**Endpoints:**
- `POST /v1/chat/completions` — Primary chat endpoint. Runs the Shared Imperator Graph and returns an OpenAI-format response.
- `GET /v1/models` — Returns all active personas from Postgres as a list of OpenAI-format "model" objects. Used by Open WebUI to populate the model picker.

**Persona resolution via `model` field:**
The `model` field in the OpenAI request body maps directly to `persona_id`. This is how the user selects which imperator to talk to from the UI.
```json
{ "model": "gunner", "messages": [{"role": "user", "content": "Hey"}] }
```
If `model` is absent or unrecognised, the graph falls back to session lookup via `active_sessions`, then defaults to `'henson'`.

**Network path:**
```
Browser → http://192.168.1.120:8340/ → henson-nginx (host port binding on m5)
Open WebUI (henson-net) → POST http://henson-langgraph:8000/v1/chat/completions
```
Nginx routes `/v1/` traffic directly to `henson-langgraph:8000`. Open WebUI's API calls are server-side and stay on henson-net — the LAN address is never used for API calls.

## 6. LangGraph Execution Engine (`henson-langgraph`)

This container is the core programmatic intelligence of Henson. It compiles and runs the Shared Imperator Graph. Phase 1 implements a simplified linear graph without advanced tool orchestration or the Brain/Voice split (which are deferred to Phase 2).

### 6.1 The Shared Imperator Concept
The graph is identical for all users and sessions. It behaves entirely based on the `persona_config` injected during the first node.

### 6.2 Graph State Definition (`GraphState`)
The heavily typed state object passed securely between nodes.

```python
from typing import Any, TypedDict, Optional

class GraphState(TypedDict):
    session_id: str
    user_input: Any                 # str for text-only; list of content parts for multimodal
    target_persona_id: Optional[str]

    # Populated dynamically by parse_and_load_persona node
    persona_config: dict            # Contains system_prompt, endpoints, temp, etc.

    # Populated by ensure_rogers_session node
    rogers_context_window_id: Optional[str]   # Rogers-issued UUID; None if Rogers unavailable

    # Populated by fetch_rogers_context node
    rogers_context_block: str       # The raw text block returned by Rogers

    # Populated by build_inference_payload
    inference_payload: dict         # The structured OpenAI-compatible request dictionary

    # Populated by execute_inference
    generated_response: str         # The final string output from the LLM

    # For graceful degradation routing
    error_state: Optional[str]
```

### 6.3 Detailed Node Specifications

#### Node 1: `parse_and_load_persona`
*   **Logic:**
    1. Checks if `target_persona_id` was explicitly passed in the state payload. If yes, updates the `active_sessions` table in Postgres via SQLAlchemy.
    2. If no, queries `active_sessions` using `session_id`. If the session is entirely new, defaults to the `'henson'` host persona.
    3. Fetches the full row from the `personas` table for the active persona.
*   **Output:** Writes the database row dictionary to `state["persona_config"]`.

#### Node 2: `ensure_rogers_session`
*   **Logic:** Ensures a Rogers conversation and context window exist for this Henson session.
    1.  Looks up `rogers_context_window_id` in `active_sessions` for this `session_id`. If found, writes it to state and continues.
    2.  If not found (first turn): calls `conv_create_conversation` to get a `conversation_id` from Rogers, then calls `conv_create_context_window` to get a `context_window_id`. Both IDs are stored back in `active_sessions.rogers_conversation_id` and `active_sessions.rogers_context_window_id`.
    3.  On Rogers failure: logs WARN, sets `rogers_context_window_id = None` in state, continues. All downstream Rogers nodes skip gracefully when `None`.
*   **Rogers calls:** `POST http://henson:9224/peer/rogers/conv_create_conversation` and `POST http://henson:9224/peer/rogers/conv_create_context_window`. Args sent directly (peer proxy handles JSON-RPC wrapping per ADR-053).
*   **`build_type_id`:** From `persona_config["rogers_assembly_flow"]`; defaults to `"standard-tiered"` if NULL (Phase 1).
*   **`max_token_limit`:** 100000 tokens (Phase 1 default).
*   **Output:** Writes Rogers-issued `context_window_id` to `state["rogers_context_window_id"]`.

#### Node 3: `fetch_rogers_context`
*   **Logic:** If `state["rogers_context_window_id"]` is None, logs WARN and sets context block to `[SYSTEM WARNING: Memory unreachable]`. Otherwise calls Rogers.
*   **Target:** `POST http://henson:9224/peer/rogers/conv_retrieve_context`
*   **Payload:** `{ "context_window_id": state["rogers_context_window_id"] }`
*   **Output:** Extracts the assembled string from the Rogers response and updates `state["rogers_context_block"]`.

#### Node 4: `build_inference_payload`
*   **Logic:** Assembles a strict OpenAI-compatible messages array. Two paths:
    *   **UI path** (when `user_input` is a full messages array — `role`/`content` dicts): Prepends the system prompt to the full history array from the client. Rogers context is NOT injected (client maintains conversation history).
    *   **MCP path** (string or multimodal content-parts array): Assembles `[system, user]` with Rogers context block prepended to user content. Multimodal content arrays are passed through to LangChain `HumanMessage` directly.
*   **Output:** Writes the constructed dictionary (including model name, temperature) to `state["inference_payload"]`.

#### Node 5: `execute_inference`
*   **Logic:** Dispatches to one of three inference paths based on `persona_config["routing_type"]`:
    1. **`peer_mad`** (Gunner): Routes via gateway peer proxy to the named MAD (`peer_mad_name`) using the MCP `chat_completions` tool — `POST http://henson:9224/peer/{peer}/chat_completions`. Full OpenAI passthrough; supports multimodal content. Per REQ-000 §7.4.1 — LangGraph never connects to `joshua-net` directly.
    2. **`openai`** (Henson): Uses `langchain_openai.ChatOpenAI`. API key read from `llm_api_key_file` at runtime. Supports multimodal via LangChain `HumanMessage` content arrays.
    3. **`gemini`** (Grace): Uses `langchain_google_genai.ChatGoogleGenerativeAI`. API key read from `llm_api_key_file` at runtime. Supports multimodal natively.
    - If `llm_api_key_file` is set, reads the API key from that file. Per REQ-000 §3.5–3.6 — no secrets in environment variables. Canonical paths: `/storage/credentials/api-keys/openai.env`, `/storage/credentials/api-keys/gemini.env`.
*   **Output:** Writes the generated text string to `state["generated_response"]`.

#### Node 6: `write_memory_to_rogers`
*   **Logic:** If `state["rogers_context_window_id"]` is None, skips silently (logs INFO). Otherwise makes two sequential calls to store the completed turn.
*   **Target:** `POST http://henson:9224/peer/rogers/conv_store_message` — same peer proxy pattern. Per ADR-053.
*   **Call 1 (user turn):** `{ "context_window_id": ..., "role": "user", "sender_id": 1000, "content": user_text }`
*   **Call 2 (assistant turn):** `{ "context_window_id": ..., "role": "assistant", "sender_id": 2033, "content": state["generated_response"] }`
    - `sender_id: 1000` = Jason (Linux OS UID 1000, the primary human operator). `sender_id: 2033` = Henson UID per registry.
    - **Phase 1 limitation:** `sender_id` for human turns is hardcoded to `1000`. Long-term, the authenticated user identity should be carried through the graph state and mapped to the user's Linux OS UID. The Linux OS UID is the canonical human identifier within the Joshua26 ecosystem — stable, system-wide, and consistent across all MADs.
    - Multimodal `user_input` is serialized to plain text (text parts joined) before storage.
*   **On failure:** Logs ERROR. Response is still delivered to the UI; the turn is lost to history.

### 6.4 Edge Routing & Graph Compilation
Because Phase 1 lacks complex tools or reflections, the edge routing is a strict linear sequence:
`START` -> `parse_and_load_persona` -> `ensure_rogers_session` -> `fetch_rogers_context` -> `build_inference_payload` -> `execute_inference` -> `write_memory_to_rogers` -> `END`.

## 7. External Dependencies & Peer Calls

Henson delegates heavy lifting to specific MADs to maintain its focus on the human interface.

### 7.1 Dependency: Rogers (Memory)
Henson relies entirely on Rogers for conversational state. It makes two specific MCP JSON-RPC calls per turn:
1.  **Retrieve Context:** `conv_retrieve_context`. Henson expects a pre-assembled text block containing summaries and recent turns.
2.  **Store Memory:** `conv_store_message`. Henson expects Rogers to manage its own background deduplication, embedding, and Mem0 extraction queues after the turn is submitted.

### 7.2 Dependency: Inference Backends (Compute)
In Phase 1, Henson acts as a standard OpenAI API client. 
- It relies entirely on the `llm_endpoint` provided by the Persona database. 
- It does not manage GPU allocations or model loading. It expects the target endpoint (e.g., `http://sutherland:11434/v1` or an external cloud API) to be "warm" and capable of answering the completion request.

### 7.3 ADR-053 Peer Proxy Compliance
Henson's LangGraph container does not connect to `joshua-net` directly. It communicates with Rogers and other internal MADs by routing requests through its own gateway's peer proxy, reachable on `henson-net`.
- **Format:** `http://henson:9224/peer/<target_mad_name>/<mcp_tool_name>` — calls go to the Henson gateway's peer proxy endpoint on `henson-net`. The gateway forwards to the target MAD on `joshua-net` via MCP JSON-RPC 2.0 (sessionless). Per ADR-053.

### 7.4 Prometheus Metrics

Henson exposes `/metrics` per [mad-prometheus-metrics-guide.md](../../guides/mad-prometheus-metrics-guide.md).

**Gateway (`henson`):** Standard metrics via `prom-client`. Labels: `mad=henson`, `tool`, `status`, `error_type`. Endpoint: `GET http://henson:9224/metrics`. (MCP traffic flows through `/mcp`; `/metrics` is a separate route.)

**MCP tool `metrics_get`:** Required for cross-network collection (joshua-net is MCP-only). Apgar calls `POST /peer/henson/metrics_get` via its gateway; that proxies to `tools/call` on this gateway. **Schema:** Input: `{}` (no args). Output: `{ "content": [ { "type": "text", "text": "<Prometheus exposition format>" } ] }` — the `text` value is identical to `GET /metrics` response. **Implementation:** Register `metrics_get` in config.json; handler invokes `registry.metrics()` (or equivalent) and returns the string. Same data source as the `/metrics` route — prom-client accumulates counters/histograms from request instrumentation.

**LangGraph (`henson-langgraph`):** Standard metrics plus Henson-specific:

| Metric | Type | Labels | Description |
|--------|------|--------|-------------|
| `henson_turn_duration_seconds` | Histogram | `persona_id`, `status` | Full conversational turn latency (parse → context → inference → store). Buckets: 1, 2, 5, 10, 30, 60, 120 |
| `henson_peer_call_duration_seconds` | Histogram | `peer`, `tool` | Rogers peer calls (conv_retrieve_context, conv_store_message). Buckets: 0.1, 0.5, 1, 2, 5 |
| `henson_inference_requests_total` | Counter | `persona_id`, `status` | Inference calls by persona (success/error) |

Expose at `GET http://henson-langgraph:8000/metrics`. Use `prometheus_client` (Python).

## 8. Nginx Proxying & UI Integration (`henson-nginx`)

Henson solves UI fragmentation by hosting the frontends. `henson-nginx` (`nginxinc/nginx-unprivileged`) listens on port `8080` within the `henson-net` boundary. It has no host port binding — all external traffic reaches it through the `henson` MCP gateway's `ui_proxy` feature. The gateway is the sole network boundary for both MCP and browser traffic. External access: `http://192.168.1.120:9224/`.

### 8.1 Core Routing Logic
```nginx
server {
    listen 8080;

    # 1. OpenAI-compatible API — routed directly to langgraph (bypasses Open WebUI)
    #    Open WebUI calls this server-side; the browser never hits /v1/ directly.
    location /v1/ {
        proxy_pass http://henson-langgraph:8000/v1/;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
    }

    # 2. Primary UI — Open WebUI (Phase 1 only)
    location / {
        proxy_pass http://henson-openwebui:8080/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "Upgrade";
        proxy_set_header Host $host;
    }

    # 3. Future: SillyTavern (add container + uncomment when needed)
    # location /tavern/ {
    #     proxy_pass http://henson-sillytavern:8000/;
    #     proxy_http_version 1.1;
    #     proxy_set_header Upgrade $http_upgrade;
    #     proxy_set_header Connection "Upgrade";
    #     proxy_set_header Host $host;
    # }
}
```

### 8.2 Adding Ephemeral UIs
New frontends can be added by spinning up a standard Docker container on `henson-net` and adding a new `location` block to the nginx config. The UI container needs no knowledge of the ecosystem — it is configured with `http://henson-langgraph:8000/v1` as its OpenAI-compatible API base URL, and `http://192.168.1.120:8340/<path>/` as its externally accessible address.

## 9. Graceful Degradation & Error Handling

Because Henson sits squarely between the Human and the Swarm, failures must be explicitly trapped and reported via the UI, rather than hanging or crashing silently.

| Dependency | Failure Mode | Circuit Breaker / Henson Behavior |
| :--- | :--- | :--- |
| **Rogers (Memory)** | `conv_retrieve` times out | Graph logs `WARN`, sets context block to `[SYSTEM WARNING: Memory unreachable]`, and proceeds to inference with zero history. |
| **Rogers (Memory)** | `conv_store` fails | Graph logs `ERROR`. The response is still delivered to the UI, but the turn is lost to history. |
| **Inference Backend** | 502 Bad Gateway / Timeout | Graph retries 3x via Tenacity. Fails over to `error_state`. Returns explicit string to UI: `[SYSTEM ERROR: Compute backend unreachable.]` |
| **henson-postgres** | Connection refused | Fatal. Graph cannot route. Gateway returns 500 JSON-RPC error. |

## 13. Phase 1.1: Compliance Fixes (2026-02-26)

Architectural review against REQ-000 identified three blockers and two deviations.
All are addressed in this section.

### 13.1 REQ 7.3 — Independent Container Startup

`henson-langgraph` previously had `depends_on: henson-postgres: condition: service_healthy`,
which blocks startup until postgres is healthy. This violates REQ-000 §7.3.

**Fix:** Remove `depends_on` from docker-compose.yml. `_startup()` in `server.py` catches
`(asyncpg.PostgresError, OSError)` and logs a warning instead of crashing. The `/health`
endpoint returns 503 when postgres is unreachable — graceful degradation at request time.

### 13.2 REQ 5.10 — Specific Exception Handling

`server.py` used blanket `except Exception as e:` throughout. REQ-000 §5.10 forbids this.

**Fix:** Each call site now catches only the exception types it can actually receive:
- Rogers calls (`_call_rogers` callers): `(httpx.TimeoutException, httpx.NetworkError, httpx.HTTPStatusError, httpx.ProtocolError, RuntimeError)`
- `_stream_peer_tokens` cleanup: `(httpx.RemoteProtocolError, httpx.TimeoutException, httpx.NetworkError, httpx.HTTPStatusError)`
- LangChain inference paths: provider exceptions imported and converted to `RuntimeError`
  before reaching the outer catch (`from openai import APIError as OpenAIAPIError`,
  `from google.api_core.exceptions import GoogleAPIError`)

### 13.3 REQ 1.10 — Unit Tests

No unit tests existed for `henson-langgraph`. REQ-000 §1.10 mandates coverage of primary
success paths and known error conditions.

**Fix:** `mads/henson/henson-langgraph/tests/` directory created with:
- `test_helpers.py` — pure function tests (`_read_credential`, `_is_messages_array`,
  `_resolve_image_urls_in_messages`)
- `test_nodes.py` — graph node tests with mocked httpx: Rogers timeout/network error
  graceful degradation, `_stream_peer_tokens` cleanup error handling after `[DONE]`,
  error propagation before `[DONE]`

Requirements: `pytest==8.3.5`, `pytest-asyncio==0.24.0`, `pytest-mock==3.14.0`

### 13.4 EX-HENSON-005 — Stream Proxy in Gateway Library

`/stream/{peer}/{apiPath}` route added to `mcp-protocol-lib/lib/server.js` for SSE
passthrough to sutherland. This transport infrastructure lives in the shared library
(same pattern as peer proxy and ui_proxy) but was previously unconditional.

**Fix:** Route is now config-gated — only active when at least one peer declares
`openai_port` in `config.json`. Registered as EX-HENSON-005 in exception registry.

### 13.5 EX-HENSON-004 — Not a Violation

Initial analysis flagged Open WebUI calling `http://henson-langgraph:8000/v1/...` directly
as a potential violation of the gateway boundary rule. **This is not a violation.**
Open WebUI is `henson-openwebui`, a container on `henson-net` — part of the Henson MAD
group. Internal communication between containers on the same private network is the intended
pattern. The gateway remains the sole boundary for traffic entering from `joshua-net`.
