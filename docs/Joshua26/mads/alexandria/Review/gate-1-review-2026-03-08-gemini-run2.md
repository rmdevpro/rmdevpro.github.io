### Gate 1: FAILED (Blockers Identified)

The design is exceptionally thorough, perfectly models the State 2 architecture (Nginx gateway + LangGraph Imperator + Backing Services), and appropriately documents necessary exceptions for infrastructure services. However, there are naming and observability convention blockers that must be addressed before implementation.

### Blockers

1. **Container Naming Convention (HLD Violation):** 
   The design names the backing services `alexandria-delphi`, `alexandria-pergamum`, and `alexandria-nineveh`. The `HLD-MAD-container-composition.md` (Design Principle 3) explicitly mandates: *"Use technology names (not generic names) for backing services to indicate official images... Backing services: `[mad]-[technology]` (e.g., `codd-postgres`)"*. 
   These containers must be renamed to reflect their underlying technologies (`alexandria-devpi`, `alexandria-verdaccio`, `alexandria-registry`), OR an explicit exception must be registered in the Exception Registry to permit the use of these flavor names.
2. **Prometheus Metric Label Mismatch:** 
   The REQ specifies `status` as a label for `mcp_requests_errors_total` (§6.1). The `HLD-MAD-container-composition.md` (Principle 3: Built-in Observability) strictly defines the labels for `mcp_requests_errors_total` as `mad`, `tool`, and `error_type`. The design must be updated to use `error_type` to ensure compatibility with Apgar's ecosystem-wide aggregations.

### Observations

1. **Storage Volume vs. Workspace Volume:** The design places the PyPI, NPM, and Docker caches in `/storage` instead of the typical `/workspace` volume (REQ-000 §3.4 usually reserves `/workspace` for MAD caches to ensure fast local I/O). However, because Alexandria serves as the centralized, permanent artifact store for the entire ecosystem and is pinned to `irina` (the ZFS storage leader), placing these caches in `/storage` is an architecturally sound choice. This observation is just to note the deliberate deviation from typical transient cache placement.
2. **Custom Image for Devpi:** The HLD assumes backing services use official images. Since there is no official image for Devpi, building a custom Python image (`alexandria-delphi`/`alexandria-devpi`) is correct and necessary, but it's worth noting that it introduces another custom image into the ecosystem to maintain.

### Recommendations

1. **Nginx Peer Proxy Configuration:** The REQ provides a summary of the nginx configuration for the `sutherland` proxy. Ensure the actual implementation includes `proxy_http_version 1.1;` and `proxy_set_header Host $host;` to correctly support SSE streaming, as explicitly outlined in the State 2 HLD.
2. **Tool Renaming:** If you rename the containers to follow the technology naming convention (e.g., to `alexandria-registry`), you should also rename the corresponding MCP tools (e.g., `alex_registry_list_images` instead of `alex_nineveh_list_images`) to maintain clarity and consistency across the ecosystem.
3. **Umask for Official Images:** The REQ correctly notes that official images (`verdaccio`, `registry`) are exempt from the `umask 000` requirement. Ensure that if they create persistent files in `/storage`, the permissions generated by their native users (like 10001 for verdaccio) are adequate for any host-level backup scripts that might need to read them.
