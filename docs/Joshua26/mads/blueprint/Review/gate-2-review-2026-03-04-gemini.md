# Gate 2 Review: blueprint

**Reviewer:** Gemini
**Gate:** 2
**MAD:** blueprint
**Date:** 2026-03-04

Gate 2: **FAILED**

**Blockers:**

1. **Offline Build Policy Violation (REQ-000 §1, REQ-blueprint §12):** The `blueprint-core` and `blueprint-claude` Dockerfiles do not implement the local package caching strategy defined in the delta REQ document (which explicitly noted an exception mapping to local `packages/` directories per ADR-037).
    - `blueprint-core/Dockerfile` runs `pip install --no-cache-dir -r /tmp/requirements.txt` instead of `--no-index --find-links=/tmp/packages/`. It is also missing the `COPY mads/blueprint/blueprint-core/packages/ /tmp/packages/` instruction.
    - `blueprint-claude/Dockerfile` runs `npm install -g @anthropic-ai/claude-code` directly from the network instead of utilizing a local npm cache directory with `--prefer-offline`.
2. **Race Condition in SSH Key Propagation (`blueprint-claude/start.sh`):** The `start.sh` script for the CLI agents attempts to copy the SSH public key from `/storage/credentials/blueprint/ssh_key.pub`. However, if the `blueprint-core` container has not yet generated this key (or is still generating it), the copy will silently fail, `authorized_keys` will be empty, and PTY connection from `blueprint-core` will be denied.

**Observations:**

1. **Directory creation in `blueprint-core/start.sh`:** The script attempts to create directories like `/storage/credentials/blueprint` and `/workspace/blueprint/bridges`. Since the container runs as a non-root user (UID 2035), these `mkdir -p` commands will crash the container with a permission denied error if the parent directories lack world-writable permissions. The README instructs the human to run `mkdir -p` manually before deployment, which is the correct and safer pattern.

**Recommendations:**

1. **Wait for SSH Key:** In `blueprint-claude/start.sh` (and presumably `blueprint-gemini/start.sh` and `blueprint-openai/start.sh`), implement a short wait loop for `/storage/credentials/blueprint/ssh_key.pub` before attempting the copy. This ensures the key generated by `blueprint-core` is available.
2. **Remove `mkdir -p` from start scripts:** Remove the `mkdir -p` commands from `blueprint-core/start.sh` to prevent potential permission-denied crashes, relying instead on the manual setup steps defined in the README.
